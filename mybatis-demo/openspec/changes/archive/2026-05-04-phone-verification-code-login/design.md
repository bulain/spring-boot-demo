## Context

当前 `/api/sys/auth/phone-login` 接口仅通过手机号查找用户，但不验证提供的验证码是否有效。项目已集成 Redisson 用于 Redis 操作。

需要添加的完整验证码流程：
1. 获取图片验证码
2. 验证图片验证码并发送短信验证码
3. 使用短信验证码登录

## Goals / Non-Goals

**Goals:**
- 实现图片验证码生成和验证
- 实现短信验证码发送和验证
- 使用 Redis 存储所有验证码（包含过期时间）
- 添加次数限制防止恶意攻击
- 保持与现有登录流程的一致性

**Non-Goals:**
- 不实现实际的短信服务商对接（仅提供接口，可后续扩展）
- 不重构现有的用户名密码登录方式
- 不添加新的用户注册流程

## Decisions

### Decision 1: 使用 Redisson 存储验证码和计数
- **Rationale**: 项目已集成 Redisson，Redis 天然适合存储具有过期时间的临时数据，且支持原子操作用于计数
- **Key**: 验证码使用 `captcha:{type}:{key}` 格式，计数使用 `captcha:count:{type}:{key}` 格式
- **过期时间**: 图片验证码 5 分钟，短信验证码 10 分钟

### Decision 2: 使用 Hutool 类库实现图片验证码
- **Rationale**: 项目已集成 Hutool，其 `cn.hutool.captcha` 包提供了成熟的验证码实现，支持干扰线、噪点等多种配置，无需自己实现图形绘制
- **实现**: 使用 Hutool 的 `LineCaptcha` 或 `CircleCaptcha`
- **长度**: 4 位随机字母数字组合
- **返回**: Base64 编码的图片

### Decision 3: 短信服务设计为可扩展接口
- **Rationale**: 不同项目可能使用不同的短信服务商（阿里云、腾讯云等），通过接口便于切换
- **默认实现**: 提供日志打印的 Mock 实现，便于开发测试

### Decision 4: 次数限制策略
- 图片验证码：同一 IP 1 分钟内最多 10 次
- 短信验证码：同一手机号 1 小时内最多 5 次，24 小时内最多 10 次
- **Rationale**: 防止短信轰炸和暴力破解，平衡用户体验和安全性

## Risks / Trade-offs

- [风险] 同一 IP 可能被多个用户共享，导致正常用户受限制 → Mitigation: IP 限制较为宽松，主要防护明显的恶意行为
- [风险] Redis 故障可能导致验证码功能不可用 → Mitigation: 属于基础设施层面问题，依赖项目已有的 Redis 高可用配置
- [风险] 图片验证码可能被 OCR 识别 → Mitigation: 可以后续添加干扰线、噪点等增强，但当前版本保持简单
