## Why

当前项目缺少统一的分布式锁抽象，业务代码直接依赖底层锁实现（如直接使用 Redisson API）导致：
1. 实现细节暴露，后续更换锁实现（如从 Redisson 迁移到其他方案）需要修改大量业务代码
2. 锁使用方式不规范，容易出现锁未释放、超时处理不当等问题
3. 缺少统一的异常处理和监控能力
4. 业务代码需要手动处理锁的获取和释放逻辑，重复代码多

## What Changes

采用**混合模式**设计，同时提供两种使用方式：

### 方式一：模板模式（推荐，类似 RedisTemplate）
- 提供 `execute()` 方法，业务代码通过回调在锁保护下执行
- 自动管理锁的获取和释放，避免资源泄漏
- 支持带超时和不带超时的重载版本
- 适合 90% 的常规业务场景

### 方式二：细粒度控制（高级使用）
- 提供 `getLock()` 方法获取锁对象，支持 try-with-resources 语法
- 支持手动 tryLock、unlock、检查锁状态等操作
- 支持可重入锁、公平锁等多种锁类型
- 适合需要自定义锁逻辑的复杂场景

### 核心特性
- 支持可重入锁和公平锁
- 支持锁等待超时、锁持有超时
- 支持 Redisson 看门狗自动续期机制
- 统一的异常处理和失败策略
- 与 Redisson 深度集成，不重复造轮子

## Capabilities

### New Capabilities
- `distributed-lock-client`: 基于 Redisson 的分布式锁客户端模板，提供统一的锁操作 API（混合模式）

### Modified Capabilities
（无）

## Impact

- **新增代码**: `com.bulain.mybatis.lock` 包下的接口、回调和实现类
- **依赖**: 使用现有 Redisson 依赖，无需新增
- **使用方式**: 业务代码通过注入 `LockClient` 使用，不直接依赖 Redisson API
- **迁移路径**: 现有代码可逐步迁移，不强制立即改造
