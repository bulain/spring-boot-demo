## Context

当前系统实现了基础的用户名密码登录，但缺少必要的安全防护机制：
- 没有密码错误次数统计和账户锁定
- 没有 IP 级别登录失败限流
- 创建用户和重置密码时没有密码复杂度校验
- 缺少防暴力破解机制
- 项目已集成 Redisson 和图片验证码功能（hutool-captcha）

## Goals / Non-Goals

**Goals:**
- 实现密码错误次数统计和账户自动锁定
- 实现登录失败 IP 限流机制
- 实现密码复杂度校验（创建用户、修改密码、重置密码）
- 实现登录图片验证码触发机制（失败 N 次后需要验证码）
- 使用 Redis 存储所有登录安全相关的计数和状态

**Non-Goals:**
- 不实现双因素认证（2FA）
- 不实现登录日志审计（可后续扩展）
- 不实现密码历史（不允许复用近期密码）

## Decisions

### Decision 1: 使用 Redis 存储安全计数
- **Rationale**: 安全计数需要高性能、支持过期、支持原子操作，Redis 完全满足
- **Key 设计**:
  - `login:fail:ip:{ip}` - IP 登录失败次数
  - `login:fail:user:{userId}` - 用户登录失败次数
  - `login:lock:user:{userId}` - 用户锁定标记
  - `login:captcha:ip:{ip}` - IP 需要验证码标记
  - `login:captcha:user:{userId}` - 用户需要验证码标记

### Decision 2: 阈值配置
- **Rationale**: 使用合理的默认值，兼顾安全性和用户体验
- **用户维度**: 连续 5 次失败触发验证码，10 次失败锁定账户 30 分钟
- **IP 维度**: 1 分钟内 10 次失败触发验证码，20 次失败限流 10 分钟

### Decision 3: 密码复杂度策略
- **Rationale**: 采用行业标准的密码复杂度要求
- **最小长度**: 8 位
- **要求**: 必须包含大小写字母、数字、特殊字符中的至少 3 种
- **禁止**: 不能包含用户名，不能是常见弱密码

### Decision 4: 验证码触发机制
- **Rationale**: 正常用户不需要验证码，可疑行为才触发
- **触发条件**: IP 或用户任一维度达到阈值即触发验证码
- **验证时机**: 登录接口先验证验证码（如果需要），再验证用户名密码
- **重置**: 登录成功后重置该用户和 IP 的所有失败计数

## Risks / Trade-offs

- [风险] 用户忘记密码导致账户被锁定 → 缓解: 提供密码重置功能，管理员可以手动解锁
- [风险] 代理环境下 IP 获取不准确 → 缓解: 使用多级 Proxy-IP 头解析，记录真实客户端 IP
- [风险] 验证码本身可能被 OCR 识别 → 缓解: hutool-captcha 已包含干扰线和噪点
- [风险] 密码复杂度要求可能影响用户体验 → 缓解: 注册/修改密码时明确提示要求
